# MCP 概念讲解稿

## 建议分成 5 层来讲

给同事介绍 MCP 时，建议不要一上来讲 SDK 或协议细节。更容易理解的顺序是：

1. 为什么需要 MCP
2. MCP 是什么
3. MCP 的核心架构
4. MCP 能暴露什么能力
5. 怎么落地使用和治理

这 5 层从业务问题讲到技术实现，适合 20 到 40 分钟的分享。

## 第 1 层：为什么需要 MCP

大模型本身只会基于上下文生成回答。它不知道公司内部系统的实时数据，也不能天然访问本地文件、数据库、业务 API 或操作工具。

过去我们给 AI 接能力，通常有几种方式：

- 在应用里硬编码 function calling。
- 给每个模型或平台单独写插件。
- 让 Agent 框架自己定义一套工具协议。
- 通过粘贴文本、导出文件等人工方式补上下文。

这些方式的问题是：

- 接入方式不统一。
- 工具很难复用。
- 客户端和服务端绑定太紧。
- 权限和审计容易散落在各处。
- 不同 AI 客户端之间迁移成本高。

MCP 要解决的核心问题就是：让 AI 应用用一种统一方式连接外部工具和数据源。

## 第 2 层：MCP 是什么

MCP，全称 Model Context Protocol，是一种开放协议，用来标准化 AI 应用和外部能力之间的连接方式。

可以把它类比成：

- USB-C：不同设备通过统一接口连接。
- LSP：编辑器通过统一协议接入不同语言服务。
- 数据库驱动：应用通过统一抽象访问不同数据库。

但 MCP 面向的是 AI 应用。它统一的是：

- AI 如何发现有哪些工具可以用。
- AI 如何调用这些工具。
- AI 如何读取外部资源。
- AI 如何使用服务端提供的提示模板。
- 客户端和服务端如何通信。

一句话总结：

> MCP 是 AI 应用连接工具、数据和上下文的开放协议。

## 第 3 层：MCP 的核心架构

MCP 是典型的 client-server 架构。

主要角色有：

- Host：用户正在使用的 AI 应用，例如 Codex、Claude Desktop、Cursor、IDE 或内部 AI 平台。
- MCP Client：Host 内部负责和 MCP Server 通信的组件。
- MCP Server：暴露工具、资源和提示词的小服务。
- 外部系统：文件系统、数据库、HTTP API、GitHub、CRM、日志系统等。

可以用这张逻辑图解释：

```text
用户
  |
  v
AI Host，例如 Codex / IDE / 桌面客户端
  |
  v
MCP Client
  |
  v
MCP Server
  |
  v
文件 / 数据库 / API / 内部系统
```

一个 Host 可以连接多个 MCP Server。比如：

- `filesystem` 负责读写指定目录。
- `github` 负责读取 issue 和 PR。
- `postgres` 负责查询数据库。
- `internal-crm` 负责查询客户信息。
- `deploy` 负责触发部署流程。

MCP Server 可以通过不同传输方式连接，常见的是：

- stdio：本地进程通信，适合本地工具。
- streamable HTTP：远程服务通信，适合部署在服务器上的 MCP 服务。

## 第 4 层：MCP 能暴露什么能力

MCP Server 主要可以暴露三类能力。

### Tools

Tools 是 AI 可以主动调用的动作。

例子：

```text
search_customer(email)
query_order_status(order_id)
create_github_issue(title, body)
run_sql(query)
send_slack_message(channel, text)
```

工具适合处理有明确输入和输出的动作。

### Resources

Resources 是 AI 可以读取的上下文。

例子：

```text
file:///project/README.md
db-schema://main/users
docs://api/payment
logs://service/orders/today
```

资源适合表示文档、文件、表结构、日志、知识库内容等。

### Prompts

Prompts 是服务端预置的提示模板。

例子：

```text
review_pr
write_release_note
generate_sql_from_question
debug_incident
```

它适合把团队内部常用流程封装起来，让不同人用同一套提示模式。

## 第 5 层：怎么落地使用和治理

落地 MCP 时，可以按成熟度分三步。

### 第一步：使用现成 MCP Server

先接入低风险、容易验证的工具：

- 文件系统，只开放测试目录。
- GitHub，只读权限。
- SQLite 或测试数据库。
- 文档知识库。

目标是让团队理解 MCP 的使用体验。

### 第二步：封装内部只读能力

把内部系统中常用查询封装成工具：

```text
get_user_profile(user_id)
get_order_status(order_id)
search_logs(service, keyword, time_range)
get_deployment_status(service)
```

这一步应该优先做只读能力，因为风险低、价值明显。

### 第三步：谨慎开放写操作

写操作要有治理机制，比如：

- 参数校验。
- 权限控制。
- 操作审计。
- 人工确认。
- 生产环境和测试环境隔离。
- 高风险动作默认禁用。

例如这些操作不应该一开始就无确认开放：

```text
delete_user
run_production_sql
restart_service
deploy_to_production
send_external_email
```

## 和普通 API 调用有什么区别

MCP Server 本质上也会调用 API、读数据库或访问文件，但它和普通 API 集成的关注点不同。

普通 API 是给应用程序调用的。MCP 是给 AI 应用发现和调用能力的。

MCP 额外强调：

- 工具描述，让模型知道什么时候该用。
- 参数 schema，让调用结构化。
- 资源发现，让上下文可枚举。
- 客户端复用，同一个 server 可以接不同 AI 客户端。
- 权限边界，让工具能力集中治理。

## 一个最小例子

我们本地这个 demo MCP Server 暴露了三个工具：

- `add`：两个数字相加。
- `current_time`：返回当前时间。
- `proof_of_call`：返回随机 nonce，并写入本地日志，用来证明 MCP 真的被调用了。

Codex 配置大概是：

```toml
[mcp_servers.demo]
command = "node"
args = ['C:\Users\admin\Desktop\mcp\dist\index.js']
```

使用时可以直接对 AI 说：

```text
请调用 demo MCP 的 proof_of_call 工具，note 写“我在验证 MCP”
```

如果工具真的被调用，本地会追加日志：

```text
C:\Users\admin\Desktop\mcp\mcp-demo-calls.log
```

这个例子能说明一件事：MCP 不只是让 AI 说话，而是让 AI 调用真实外部能力。

## 分享时可以强调的结论

- MCP 是 AI 应用连接外部工具和数据源的开放协议。
- 它的价值是统一接入方式，让工具可以被不同 AI 客户端复用。
- MCP 最重要的抽象是 Tools、Resources、Prompts。
- 落地时应该先从只读、低风险、可审计的能力开始。
- 对企业来说，MCP 的关键不是炫技，而是把内部系统能力安全地暴露给 AI 助手。

## MCP 可能的不足和风险

MCP 很有价值，但它不是银弹。团队落地时需要提前理解它的边界。

### 1. 协议和生态还在快速变化

MCP 目前更像 AI 工具生态里的事实标准，而不是像 HTTP、SQL 那样长期稳定的基础标准。

这意味着：

- 协议版本可能继续演进。
- SDK API 可能变化。
- 不同客户端的支持程度可能不一致。
- 一些能力在不同 Host 中表现不完全相同。

所以在生产环境中使用 MCP，要关注版本锁定、升级策略和兼容性测试。

### 2. 客户端兼容性不一定完全一致

理论上，一个 MCP Server 可以被多个 MCP Client 使用。但实际情况中，不同客户端可能在以下方面存在差异：

- 支持的传输方式不同。
- 工具展示方式不同。
- 用户确认流程不同。
- 对资源和 prompts 的支持程度不同。
- 日志、权限、错误处理体验不同。

因此，不应该默认“写一次 MCP Server，所有客户端都能完美使用”。更现实的做法是选定主要客户端进行测试。

### 3. 安全边界容易被低估

MCP Server 往往连接真实系统，例如文件、数据库、内部 API、部署系统。它一旦设计不好，就可能把高风险能力暴露给 AI。

典型风险包括：

- 工具权限过大。
- 参数校验不严格。
- 允许模型执行任意 SQL 或 shell 命令。
- 缺少审计日志。
- 生产和测试环境隔离不足。
- 高风险写操作没有人工确认。
- 工具描述写得太宽泛，导致模型误用。

MCP 的安全治理应该按“最小权限”设计，而不是把内部 API 原样暴露出去。

### 4. 工具质量会直接影响 AI 质量

MCP 只是连接协议，不会自动让工具变好。

如果工具设计得不好，AI 仍然会用得很差。例如：

- 工具名称不清楚。
- 描述不准确。
- 参数 schema 太宽泛。
- 返回结果太长或结构混乱。
- 错误信息不可读。
- 工具粒度过粗或过细。

好的 MCP Server 需要像设计产品 API 一样设计工具接口。

### 5. 可观测性和调试成本会上升

引入 MCP 后，一次 AI 响应可能涉及：

```text
用户问题
  -> 模型推理
  -> 工具选择
  -> MCP Client
  -> MCP Server
  -> 外部系统
  -> 结果返回
  -> 模型组织回答
```

任何一环出问题，用户看到的都可能只是“回答不对”或“工具失败”。

因此需要配套：

- 工具调用日志。
- 输入输出审计。
- 错误码和错误信息规范。
- 延迟统计。
- 权限拒绝记录。
- 关键操作 trace id。

否则排查问题会比较困难。

### 6. 成本和延迟需要控制

MCP 调用外部系统会增加响应时间。如果一个任务需要连续调用多个工具，整体延迟会变明显。

同时，工具返回大量上下文也会增加模型 token 消耗。

常见优化方式包括：

- 工具返回结构化、精简结果。
- 分页或限制返回条数。
- 给工具增加筛选参数。
- 对常用只读结果做缓存。
- 避免把整份大文档或大日志一次性塞给模型。

### 7. 不能替代业务权限系统

MCP 应该接入现有权限体系，而不是绕过它。

例如用户在 CRM 中没有权限查看某客户信息，那么通过 MCP 也不应该能看到。MCP Server 应该识别调用者身份，并在服务端执行权限判断。

如果做不到身份透传，至少也要把 MCP Server 运行在受限账号下，只开放明确允许的数据和动作。

## MCP 的前景

总体判断：MCP 很可能会成为 AI 应用连接外部工具和数据源的重要协议之一，尤其适合开发工具、企业内部系统和个人自动化场景。

### 1. 它解决的是一个真实痛点

AI 应用要真正进入工作流，必须连接外部上下文和工具。

没有 MCP 或类似协议时，每个客户端、每个工具、每个 Agent 框架都要重复造轮子。MCP 把这部分抽象出来，让工具提供方和客户端提供方可以解耦。

这类“连接层标准”一旦被足够多工具采用，网络效应会比较强。

### 2. 开发工具场景会最先成熟

MCP 与开发者工作流天然契合，因为开发者环境里有大量可工具化的东西：

- 代码仓库。
- 文件系统。
- Git。
- GitHub/GitLab。
- 数据库。
- CI/CD。
- 日志和监控。
- 文档系统。
- 本地脚本。

所以 IDE、代码 Agent、命令行 AI 工具会是 MCP 的高频使用场景。

### 3. 企业内部系统会有很大空间

企业内部往往有大量系统和数据，但 AI 很难直接使用。

MCP 可以把这些能力包装成受控工具：

- 查询客户。
- 查询订单。
- 查询库存。
- 查询部署状态。
- 搜索知识库。
- 分析工单。
- 生成报表。

长期看，企业可能会建设自己的 MCP Server 层，把内部能力统一暴露给多个 AI 助手。

### 4. MCP Server 可能成为新的集成资产

过去企业会沉淀 API、SDK、CLI、内部平台。未来可能还会沉淀 MCP Server。

一个设计良好的 MCP Server 可以同时服务：

- Codex。
- IDE 插件。
- 内部客服助手。
- 运维助手。
- 数据分析助手。
- 自动化 Agent。

这会让 MCP Server 成为一种可复用的集成资产。

### 5. 安全和治理产品会跟着出现

只要 MCP 进入企业环境，就会需要配套治理能力。

可能出现的方向包括：

- MCP 网关。
- 工具权限管理。
- 工具调用审计。
- 高风险操作审批。
- 敏感数据脱敏。
- 工具市场和版本管理。
- 企业级 MCP Server 注册中心。

这说明 MCP 不只是一个开发者协议，也可能形成一层企业 AI 基础设施。

### 6. 仍然可能与其他方案长期共存

MCP 不一定会消灭其他工具调用方式。

未来很可能是多种方式共存：

- 模型厂商自己的 tools/function calling。
- 应用内部的专用工具系统。
- Agent 框架自己的 tool abstraction。
- MCP 作为跨客户端、跨工具的开放连接层。

MCP 的优势在于开放和复用，但在一些强绑定平台内，原生工具调用仍然会长期存在。

## 对团队的建议

短期建议：

- 先学习 MCP 的概念和工具设计方法。
- 用低风险 demo 熟悉调用链路。
- 优先封装只读能力。
- 避免一开始接生产写操作。

中期建议：

- 为内部常用系统设计 MCP Server。
- 建立工具命名、schema、错误返回规范。
- 加入日志、审计和权限控制。
- 明确哪些 MCP Server 可以用于生产环境。

长期建议：

- 把 MCP 当作企业 AI 工具接入层的一种候选标准。
- 将 MCP Server 纳入内部平台治理。
- 关注协议版本和主流客户端支持情况。
- 对高价值场景做深，而不是盲目接入所有系统。

## 一句话收尾

> MCP 让 AI 从“只能回答问题”变成“可以在受控边界内使用工具和上下文完成任务”。
